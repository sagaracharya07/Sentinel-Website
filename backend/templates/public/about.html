About the Project

Sentinel AI is a final-year computing project

Built for NIT3004 IT Capstone Project 2 at Victoria University. Presented honestly — as a working prototype, not a commercial product.

The problem

Phishing remains one of the most common paths to account compromise and data breach, and most organisations rely on generic spam filtering with little visibility into why a message was flagged, or a structured way for employees to report what they see.

Objectives

  • Connect directly to a real Gmail mailbox via Google OAuth and the Gmail API
  • Classify incoming mail using a trained machine-learning model, not fixed rules alone
  • Explain every verdict with structured, human-readable findings
  • Give administrators real quarantine/release control without ever deleting mail
  • Give employees a simple way to report suspicious email and track its outcome

Architecture

A Flask + Jinja2 backend with SQLAlchemy models, Alembic migrations, and Celery/Redis for background processing (Gmail sync, model retraining, scheduled tasks). PostgreSQL in production, SQLite for local development.

The machine-learning model's role

A trained classifier estimates a phishing probability from message content and structure. That probability, combined with structured security findings (authentication, sender identity, links, attachments), produces the operational verdict. The project model achieved approximately 96.1% accuracy on its held-out evaluation dataset (v1, 7,539 test messages).

Gmail integration

Google OAuth for connection, the Gmail API for retrieval and label operations, and either polling or optional Pub/Sub push for detecting new mail.

Human review

No classifier is perfect. The "Needs Review" state exists specifically for uncertain cases, and every automated verdict can be confirmed, released, or overridden by an administrator — with the override recorded in the audit log.

Technology stack

Python / Flask, SQLAlchemy, Alembic, Celery, Redis, PostgreSQL, scikit-learn, and a hand-built HTML/CSS/JavaScript front end (no SPA framework).

Testing

264 automated tests cover authentication, CSRF protection, the classifier, Gmail OAuth/sync/labels/parsing, .eml reporting, model governance, and access control.

Security principles

Encrypted OAuth tokens, CSRF protection on every state-changing request, role-based access control, rate limiting, security headers, and an append-only audit log.

Known limitations

  • Single-mailbox prototype — one connected Gmail account per deployment
  • No independent DNS validation of SPF/DKIM/DMARC beyond the provider's reported results
  • Related-message grouping is basic (sender/domain), not a full correlation engine
  • No production security audit or compliance certification

Future development

Microsoft 365 / Graph integration, richer campaign correlation, and expanded notification channels are natural next steps beyond this project's current scope.
