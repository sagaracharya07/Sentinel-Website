Security

What's actually implemented, not just claimed.

For a platform whose entire purpose is spotting security threats, it matters that its own security practices are real and checkable — so here's what's actually running, split from what's a deployment-environment responsibility.

Authentication & access control

  • Passwords are hashed with Werkzeug's PBKDF2 implementation — never stored or logged in plaintext.
  • Sessions use signed, HttpOnly, SameSite cookies; the session secret must be a real, unique value in production or the application refuses to start.
  • Roles (user/admin) are enforced server-side on every request. There is no client-side role selector — a request either has the right session or it doesn't.
  • Self-registration can only ever create a standard user account; administrator access is never client-assignable.
  • Login, registration, password reset, and the contact form are all rate-limited to slow down abuse.

Data handling

  • Email bodies are automatically redacted 24 hours after a scan by a real background job — not a policy described in a document.
  • Every scan, login, admin action, retrain, and mailbox sync is written to an audit log.
  • Mailbox access is read-mostly: the only write actions are a reversible quarantine-folder move or an in-place flag, never a permanent delete.

Application security

  • A Content-Security-Policy and Strict-Transport-Security header are set on every response.
  • Password-reset and email-verification tokens are single-use, time-limited, and generated with a cryptographically secure random source.
  • Password-reset requests return an identical response whether or not an email is registered, to prevent account enumeration.

Deployment-environment responsibilities

TLS/HTTPS termination is handled by whatever reverse proxy or hosting platform sits in front of the application (Render, nginx, Caddy, etc.) — this is the standard pattern for an app built this way, and is called out here explicitly rather than faked with a self-signed certificate.
